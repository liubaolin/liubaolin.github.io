tcpdump-抓包

发表于 2021-11-07 | 分类于 网络协议 | | 阅读次数:

01 简介

tcpdump 是一款 Linux 平台的抓包工具。它可以抓取涵盖整个 TCP/IP 协议族的数据包，支持针对网络层、协议、主机、端口的过滤，并提供 and、or、not 等逻辑语句来过滤无用的信息。

tcpdump 是一个非常复杂的工具，掌握它的方方面面实属不易，也不推荐，能够用它来解决日常工作问题才是关系。

02 tcpdump 命令选项

tcpdump 有很多命令选项，想了解所有选项可以 Linux 命令行输入 tcpdump -h，man tcpdump 查看每个选项的意思。

下面列举一些常用选项：

  • -A 只使用 ASCII 打印报文的全部数据，不要和 -X 一起使用，获取 http 可以用 tcpdump -nSA port 80
  • -b 在数据链路层上选择协议，包括 ip, arp, rarp, ipx 等
  • -c 指定要抓取包的数量
  • -D 列出操作系统所有可以用于抓包的接口
  • -e 输出链路层报头
  • -i 指定监听的网卡，-i any 显示所有网卡
  • -n 表示不解析主机名，直接用 IP 显示，默认是用 hostname 显示
  • -nn 表示不解析主机名和端口，直接用端口号显示，默认显示是端口号对应的服务名
  • -p 关闭接口的混杂模式
  • -P 指定抓取的包是流入的包还是流出的，可以指定参数 in, out, inout 等，默认是 inout
  • -q 快速打印输出，即只输出少量的协议相关信息
  • -s len 设置要抓取数据包长度为 len，默认只会截取前 96bytes 的内容，-s 0 的话，会截取全部内容。
  • -S 将 TCP 的序列号以绝对值形式输出，而不是相对值
  • -t 不要打印时间戳
  • -vv 输出详细信息（比如 tos、ttl、checksum等）
  • -X 同时用 hex 和 ascii 显示报文内容
  • -XX 同 -X，但同时显示以太网头部

03 过滤器

网络报文是很多的，很多时候我们在主机上抓包，会抓到很多我们并不关心的无用包，然后要从这些包里面去找我们需要的信息，无疑是一件费时费力的事情，tcpdump 提供了灵活的语法可以精确获取我们关心的数据，这些语法说得专业点就是过滤器。

过滤器简单可分为三类：协议（proto）、传输方向（dir）和类型（type）。

一般的 表达式格式 为：

  • 关于 proto：可选有 ip, arp, rarp, tcp, udp, icmp, ether 等，默认是所有协议的包
  • 关于 dir：可选有 src, dst, src or dst, src and dst，默认为 src or dst
  • 关于 type：可选有 host, net, port, portrange（端口范围，比如 21-42），默认为 host

05 与 wireshark、Snort 等工具的结合

tcpdump 抓包的时候，默认是打印到屏幕输出，如果是抓取包少还好，如果包很多，很多行数据，刷刷刷从眼前一闪而过，根本来不及看清内容。不过，tcpdump 提供了将抓取的数据保存到文件的功能，查看文件就方便分析多了，而且还能与其他图形工具一起配合分析，比如 wireshark、Snort 等。

  • -w 选项表示把数据报文输出到文件
  • -r 选项表示读取文件里的数据报文，显示到屏幕上

PS：.pcap 格式的文件需要用 wireshark、Snort 等工具查看，使用 vim 或 cat 会出现乱码。

Flink03-Flink流处理基础

发表于 2021-06-12 | 分类于 数据处理 | | 阅读次数:

时间

事件时间（event time）

事件发生的顺序，这个顺序是不会变的，计算结果是可复现的

处理时间（processing time）

基于Flink部署的本地机器时间,每次可能不一样，不提供确定性的保障

水位线（Watermark）

在使用Event Time时间概念时，Flink需要知道：

  1. 每个StreamElement的EventTime时间
  2. 接入的数据何时可以开始计算（什么时间算结束）
  • Watermark用于标记Event-Time的前进进程；
  • Watermark跟随DataStream Event-Time变动，并自身携带TimeStamp；
  • Watermark用于表明所有某个（较早）时间点之前的事件（可能）已经到达；
  • Watermark本身也属于特殊事件

数据流事件本身有顺序的情况下，可以得到完美的Watermark。

当数据流事件无序的时候，通常认为它们有一定的无序性，可以通过设置maxOutOfOrderness最大时延要求来产生Watermark，并且只有新的最大时间戳出现时，
才会产生新的Watermark

如果消息的时间小于等于当前Watermark的时间，则该事件会被标记为迟到事件,不会被纳入窗口计算

Wartermark与window之间的关系

wartermark大于等于window结束时间会触发统计

对于事件时间小于wartermark时间的事件，不会纳入相关窗口的统计

Wartermark使用总结

  • Watermark = Max EventTime = Late Threshold （接入数据最大等待时延的容忍）
  • Late Threshold越高，数据处理延迟越高
  • 解决一定范围内的乱序事件
  • 窗口触发条件：Current Watermark > Window EndTime
  • Watermark的主要目的是告诉窗口不会再有比当前Watermark更晚的数据到达
  • idel Watermark（理想的Watermark）仅会发生在顺序事件中

如何定义和使用Watermark

两种策略

  • Periodic Watermarks

基于事件时间和最大时延

  • Punctuated Watermarks

基于固定的事件，在事件流里通过固定的信息标记

上面都是通过外部的方式去DataStream API里面去指定EventTime的抽取逻辑以及Watermark的生成逻辑。
对于大部分消息中间件来讲，比如kafka，可以在生成消息的时候就把Watermrk和Timestamp生成和抽取出来.

在Source Functions中定义Timestamps和Watermarks

1.11版本新的设置Timestamp和Watermark的api：Watermark Strategies

window 窗口计算

窗口将无界的数据流，分隔成有界的数据（按时间或数量）进行计算

窗口计算类型：

  • 聚合统计

  • 记录合并

  • 双流join

Window Assigner

Flink窗口骨架结构中有两个必须的操作：

  1. 使用窗口分配器（WindowAssigner）将数据流中的元素分配到对应的窗口

  2. 当满足窗口触发条件后，对窗口内的数据使用窗口处理函数（Window Function）进行处理，常用的Window Function有reduce、aggregate、process。

Flink支持的窗口类型：

  1. time window
  • Tumbling Windows（滚动窗口）
  • Sliding Windows（滑动窗口）
  • Session Windows（会话窗口）
  • Global Windows（全局窗口）
  1. count window
  • 滚动count
  • 滑动count

所有的窗口类型都分为Keyed Windows和Non-Keyed Windows

Window Trigger

窗口触发器决定了何时启动Window Function来处理Window中的数据，以及何时清理Window中的数据

  • 通常情况下WindowAssigner中已经默认包含相应的Windwo Trigger，且适用于大部分场景
  • 所有基于Event Time时间的WindowAssigner默认Trigger都是EventTimeTrigger
  • GlobalWindow默认为NeverTrigger，也就是不触发，因此使用GlobalWindow必须指定CustomTrigger，否则窗口永远不触发
  • 通过WindowStream提供的trgger（）方法能够接受新的Trigger实现，但是会覆盖WindowAssigner自带的默认Trigger

Window Evictors

数据剔除器（可选组件），决定了数据清理策略。

Flink内置了几个数据剔除器

  • CountEvictor
  • TimeEvictor
  • DeltaEvictor

Window Function

增量计算（会维护状态信息，基于状态的统计）

  • ReduceFunction
  • AggregateFunction
  • Foldfunction

全量计算

  • ProcessWindowFunction
    对窗口内所有的数据进行计算，会带来比较大的性能上的开销

Window 多流合并

Window Join

必须共用一个窗口，窗口类型必须统一。

  • Tumbling Window Join
  • Sliding Window Join
  • Session Window Join

    Interval Join

在Flink DataStream API中使用

  • ProcessJoinFunction

Process Function应用

ProcessFunction函数是低阶流处理算子

  • 事件（数据流元素）
  • 状态（容错和一致性）
  • 定时器（事件时间和处理时间)

总结：

对于每一个接入的数据元素：

  • 更新数据状态
  • 注册未来某一时间需要调用的callback回调函数

当某一时间来到后：

  • 检查是否满足条件，并执行对应的行为，例如输出数据元素等

SideOutput旁路输出

按照给定的数据切分逻辑，将输入数据切分成不同的子集

Asynchronous I/O异步操作

Stream Enrichment Pattern 对流处理过程的数据进行补充丰富，以提供更多的指标计算维度

外部信息特点：

  • 数据量巨大（亿级别）
  • 通常维护和存储在外部的服务或者DB中
  • 单条查询有一定的时延（ms/s级别）

可以通过算子中直接关联数据库的方式对数据进行补充，但是存在性能瓶颈，如果查询缓慢会导致整个算子的执行效率降低，进而导致资源利用率下降，并发的Task都会被block住。

数据库连接会严重控制和影响整个应用的throughput和latency。

可以通过异步的方式，并行的将结果取回来并缓存。Flink可以通过异步的方式同数据库进行交互。

异步交互前提条件是数据库的客户端要支持异步请求。

AsyncFunction API调用

自定义SourceFunction

通过实现SourceFunctin接口来实现对外部数据源的读取，并转换成对于的DataStream数据集

Flink01-集群运行模式

发表于 2021-06-05 | 分类于 数据处理 | | 阅读次数:

Session Mode

JobManager和TaskManager共享

优点

  • 资源充分共享，提示资源利用率
  • Job在Flink Session集群中管理，运维简单

缺点

  • 资源隔离性相对较差（job资源不相互隔离）
  • 非Native部署，TM不易扩展(TM数量在启动时已经固定并启动起来了)，Slot计算资源伸缩性较差

Per-Job Mode

  • 单个Job独享JobManager和TaskManager
  • TM中的Slot资源根据Job指定

优点

  • Job与Job之间的资源充分隔离
  • 资源根据Job需要进行申请，TM Slots数量可以不同

缺点

  • 资源相对比较浪费，Job Manager需要消耗资源
  • Job管理完全交给ClusterManagement，管理复杂

Application Mode（1.11版本提出）

优点

  • 有效的降低宽带和客户端负载
  • Application实现资源隔离，Application中实现资源共享

缺点

  • 仅支持Yarn和K8s部署

Kubernetes02-核心概念

发表于 2021-05-12 | 分类于 SRE | | 阅读次数:

比较容器和虚拟机

和虚拟机相比容器更加轻量级，因为每个虚拟机需要运行自己的一组系统进程，而容器只是运行在宿主机上被隔离的单个进程。

虚拟机的好处是它们提供完全隔离的环境，因为每个虚拟机都运行在它们自己的Linux内核上，而容器则是调用同一个内核。

那容器是如何实现隔离的呢？有两个机制可用：

  1. Linux命名空间
    它使每个进程只看到自己的系统视图（文件、进程、网络端口、主机名等）
  2. Linux控制组（cgroups）
    它限制了进程能使用的资源（cpu、内存、网络带宽等）

Pod

Kubernetes中的基本构成模块,基本操作单元。
一个pod就是一组紧密管理的容器（也可能是一个）,它们一起运行在一个工作节点上，及同一个Linux命名空间中。
一个pod就像一个独立的逻辑机器，它有独立的IP、主机名、进程等。

区别于容器

容器被设计成每个容器中只能运行一个进程，pod中可以运行多个容器（便于相关服务部署在一起）。

容器间彼此完全隔离，pod中的容器可以共享相同的命名空间，可以共享相同的主机名和端口,同样的这些容器在相同的IPC命名空间下运行，因此可以使用IPC进行通信。

可以通过localhost与同一个pod中的其它容器进行通讯。

平坦pod间网络

Kubernetes集群中的所有pod都是在同一个共享网络地址空间中。所以每个pod都可以通过访问其它pod 的ip地址实现相互访问。（它们之间没有NAT）

yaml文件重要参数

  • metadata
    包括名称、命名空间、标签和关于容器的其它信息
  • spec
    包括pod的实际说明， 例如pod的容器、卷和其它数据
  • status
    运行中的pod的当前信息，例如pod所处的条件、每个容器的描述和状态，以及内部ip和其它信息

需要创建服务对象公开pod，服务对象是一个特殊的LoadBalanceer类型的服务，可以通过服务对外共该的ip访问pod。

服务

pod通常需要对来自集群内部的其它pod，以及来自集群外部的http请求作出响应。

pod需要一种寻找其它pod的方法来使用其它pod提供的服务，service就是用来做这件事的。

service服务是一种为一组功能相同的pod提供单一不变接入点的资源，当服务存在时它的ip和端口不会变。

服务对pod的请求是负载均衡的。

pod代表Kubernetes中最小的部署单元，在实际场景，我们希望部署的资源可以自动保持运行，并且保持健康，要做到这一点我们几乎不会直接创建pod
，而是创建ReplicaSet（老版本是ReplicationController）或Deployment这样的资源，接着由它们来创建并管理实际的pod

存活探针

Kubernetes可以通过存活探针（liveness probe）检查容器是否还在运行。可以为pod中的每个容器单独指定探针，Kubernetes会定期执行探针，如果探测失败，将重启容器。

三种探针

  • HTTP GET探针对容器的IP地址（指定端口和路径）执行HTTP GET请求。
  • TCP套接字探针尝试与容器指定端口建立TCP连接。如果连接建立成功，则探测成功。
  • Exec 探针在容器内执行任意命令，并检查命令的退出状态码，如果状态码是0，则探测成功。

创建基于HTTP的存活探针

这里就定义了一个http get存活探针

kubia-unhealthy服务健康检查每调用5次就会失败一次，所以这里过2分钟会发现服务被重启过一次了。
通过kubectl describe pod 命令可以看到容器当期的状态，重启了多少此，上次被终止的原因等等。

我们还可以配置探针的delay（延迟）、timeout（超时）、period（周期）等,可以通过kubectl explain命令查看yaml详细配置说明

  • initialDelaySeconds 表示容器启动后延迟多久进行检测，默认0，及容器启动后即开始检测。
  • timeoutSeconds 表示超时时间，超时未响应则检测失败
  • periodSeconds 表示多久检测一次

在生成环境中运行的pod一定要定义一个存活探针。没有探针的话Kubernetes无法知道你的应用是否存活。

就绪探针

与存活探针不同，如果容器没有通过准备检查，不会被终止或重启，但是该容器将从endpoint中移除，也就是不会有流量发送到改pod,注意当检测到pod恢复正常后会被重新添加到endpoint中。
就绪指针确保有问题的pod不会被访问到。

ReplicaSet

ReplicaSet是一种Kubernetes资源，它可以确保pod始终保持运行状态，如果pod因为任何原因消失（比如节点从集群中消失，或者pod从节点中驱逐），
RS会注意缺少了pod，并创建代替。

一般而言RS旨在创建和管理一个pod 的多个副本，RS的工作是确保标签的数量始终与其标签选择器匹配，如果不匹配，rs将根据所需，采取适当的操作来协调pod 的数量。

RS由3部份组成

  • label selector 标签选择器，用于确认rs作用域中有哪些pod
  • replica count 副本个数，指定运行的pod数量
  • pod template pod模板，用于创建新的pod副本

RS可以轻松实现pod的水平伸缩
